Standard client-work lifecycle

From request to documented delivery.

Krahnie’s job is to keep work moving through a clear, observable, repeatable sequence while humans do the Salesforce work first and the agent handles orchestration, metadata capture, PRs, CI/CD tracking, and documentation.

1
Client request intake

A request from Slack, email, meeting notes, client portal, manual entry, or webhook is captured. Krahnie extracts client, org/project, urgency, acceptance criteria, and unknowns.

2
Card creation and identity mapping

A delivery card is created or linked under the right client/org/project. Requester identity maps to GitHub/Jira/Linear/Asana identity when available, so ownership and audit trails stay clean.

3
Estimate, due date, assignment

Krahnie proposes complexity, effort, due date, assignee, reviewer, likely impacted Salesforce areas, and open questions. Humans approve or edit on the card.

4
Human-first Salesforce work

The assigned admin/developer does the work in Salesforce Setup, a dev box, sandbox, Salesforce CLI, or local metadata repo. Krahnie keeps the card lane ready without trying to own every implementation step.

5
Ready for Customer Review

When the human finishes, moving the card to Ready for Customer Review tells Krahnie the work is ready to package, review, and deploy for testing.

6
Metadata capture

Krahnie uses manually supplied metadata notes or polls/retrieves recent changes from the dev box via Salesforce CLI, so the human does not need to list every XML file by hand.

7
Branch and commit

Krahnie creates the branch/worktree as needed, adds retrieved metadata, commits with card context, and links the work back to the request.

8
Pull request

PR templates include intent, changed metadata, customer-review notes, deployment plan, risks, rollback notes, and documentation placeholders.

9
Review and merge to partial branch

Reviewers get a concise summary and diff highlights. Approved changes merge to a connected partial branch, not directly to production.

10
Standard CI/CD deployment

CI detects changed Salesforce metadata and deploys the relevant delta to the connected testing environment. Deployment logs return to the card and PR.

11
Ready for Customer Testing

After deploy success, the card moves to customer testing. Krahnie tracks automated tests, smoke checks, admin QA steps, user acceptance notes, screenshots, and known limitations.

12
Accepted / Waiting Deployment documentation

When the change is accepted, Krahnie drafts functional docs explaining what changed and technical docs explaining metadata, integration impact, dependencies, and rollback path.

13
Feedback and lessons loop

After delivery, Krahnie captures what surprised the team, client-specific quirks, process improvements, reusable prompts, and runbook updates.

Request thread/card pattern

Every request gets a canonical card, regardless of source. If there is a thread or conversation, Krahnie mirrors important state both ways: owner, due date, branch, PR, deploy status, validation evidence, doc links, and final outcome.
